# 定价模型——深度解析

SaaS 定价模型综合参考，含真实案例和适用场景分析。

---

## 模型一：按席位/按用户

**运作方式：** 价格 × 访问产品的用户数。

**适用场景：**
- 用户越多价值越大的协作工具
- 每个销售代表都需要使用的 CRM
- 以组织为购买方、席位与人数对应的工具

**案例：** Salesforce（$25-300/席位/月）、Linear（$8/席位/月）、Figma（$12/席位/月）、Notion（$8/席位/月）

**扩展机制：** 随公司招聘自动增长。无需追加销售对话——新员工开通席位，收入自然增长。

**失败模式：**
- 单一核心用户工具（一个人完成所有工作，团队只看结果）→ 按席位计费惩罚了客户
- 承包商或外部利益相关者使用的工具 → 计费变成一场谈判
- 共享账号容易且难以阻止的产品

**席位定价变体：**

| 变体 | 说明 | 案例 |
|------|------|------|
| 指定席位 | 每个许可对应特定用户 | Salesforce |
| 并发席位 | 同时最多 N 个用户在线 | 传统企业软件 |
| 创建者/查看者分离 | 创建者付费，查看者免费或低价 | Figma、Miro |
| 最低席位数 | 套餐要求最少 X 个席位 | 大多数企业交易 |

**提示：** 创建者/查看者定价非常适合一个团队创建、几十人查看的 B2B 工具。它既能驱动传播（免费查看者），又能从实际使用者那里获取收入。

## 模型二：按用量（消耗制）

**运作方式：** 客户为实际使用量付费——API 调用次数、存储空间、计算资源、发送的消息、投递的邮件。

**适用场景：**
- 基础设施和开发者工具
- 计算成本随用量增长的 AI/ML 工具
- 通信平台（邮件、短信、视频）
- 客户之间使用量差异极大的产品

**案例：** Stripe（2.9% + $0.30/笔交易）、Twilio（$0.0075/条短信）、AWS（按具体服务计费）、各大 AI 模型服务（按 token 计费）

**扩展机制：** 自然增长——客户业务增长，使用量增长，收入随之增长，无需任何主动操作。SaaS 中最优的获客成本与客户终身价值比。

**失败模式：**
- 不可预测的账单 → 客户主动限制用量避免超额 → 你自己给自己设了天花板
- 经济下行时高流失 → 使用量下降，收入同步下降
- 对你和客户来说都难以预测

**用量定价变体：**

| 变体 | 说明 | 案例 |
|------|------|------|
| 纯消耗制 | 只为实际用量付费 | AWS Lambda |
| 预付积分 | 先购买积分，按需消耗 | 各大 AI 平台、Resend |
| 承诺用量 + 超额 | 固定费用含用量上限，超出按量计费 | Stripe、Twilio 批量套餐 |
| 阶梯用量 | 用量越大，单价越低 | Mailchimp 邮件阶梯 |

**混合方式：** 大多数成熟的按用量计费公司会增加平台费（小额月固定费用），确保收入底线，减少低使用量月份的流失。

## 模型三：按功能（阶梯式固定费用）

**运作方式：** 不同功能包对应不同的固定价格。即基础-进阶-高级模型。

**适用场景：**
- 不同客户群体之间功能需求差异明显的产品
- 可预测支出很重要的市场（CFO 喜欢这种方式）
- 从中小企业到大企业，企业级功能确实有本质差异的产品

**案例：** HubSpot（入门版/专业版/企业版）、Intercom（入门版/专业版/高级版）、大多数 SaaS 产品

**扩展机制：** 需要追加销售动作——客户需要超出当前层级的功能才会升级。不如按用量计费自动化，但更可预测。

**失败模式：**
- 功能层级与实际客户需求不匹配 → 客户集中在一个层级，无人升级
- 企业功能不够有吸引力，无法支撑价格跳跃 → 卡在中端市场
- 层级太多 → 选择困难

---

## 模型四：固定费用

**运作方式：** 一个价格，包含所有功能，无限使用。

**适用场景：**
- 成本结构可预测的小型工具
- 以简单为核心差异化的市场
- 使用量客户之间差异不大的产品

**案例：** Basecamp（$99/月固定价）、Transistor.fm（按播客数计费，不限听众）、Calendly 基础版

**扩展机制：** 没有。你需要一个高级套餐或附加组件，否则只能依靠获取新客户。

**失败模式：**
- 重度用户被轻度用户补贴 → 重度用户长期留存，轻度用户流失 → 逆向选择
- 无法从现有客户获取更多收入 → 除非增加层级或涨价，否则增长受限

**固定费用适用条件：** 当你的服务成本确实是固定的，或者"简单"这一市场定位的价值超过按用量计费能获取的额外收入。

## 模型五：免费增值（Freemium）

**注意：** 免费增值是一种获客策略，而非定价模型。它可以与以上任何模型组合使用。

**运作方式：** 功能受限的免费层级 + 付费层级。

**适用场景：**
- 开发者工具（产品驱动增长 PLG）
- 能病毒式传播的协作工具
- 用户越多价值越大的网络效应产品

**案例：** Slack、Notion、Figma、GitHub、Airtable

**免费增值的数学逻辑：**
- 免费用户有服务成本
- 付费转化率必须高到能覆盖免费用户的成本
- 经验法则：2-5% 的免费转付费率在规模化后可行，1-2% 通常不行

**免费版 vs 试用 vs 免费增值：**

| 模式 | 说明 | 适用场景 |
|------|------|---------|
| 永久免费层级 | 功能永久受限的免费套餐 | PLG、病毒式传播 |
| 限时试用 | 14-30 天完整功能访问 | 销售驱动、复杂产品 |
| 限量试用 | 完整功能，用完即止 | 开发者工具、AI |
| 免费增值 | 永久受限，引导升级付费 | 自下而上的企业渗透 |

## 模型六：混合定价

大多数成熟的 SaaS 公司最终都会采用混合定价。常见组合：

| 组合 | 案例 |
|------|------|
| 平台费 + 按席位 | 基础访问权 + 用户许可证 |
| 平台费 + 按用量 | 月度最低消费 + 超额部分 |
| 功能层级 + 用量 | 套餐决定包含用量，超出部分另计 |
| 按席位 + 按用量 | 席位许可 + 重度用户的用量计费 |

**何时选择混合定价：**
- 你同时有固定基础设施成本和可变服务成本
- 你需要收入底线（平台费）+ 增长空间（用量）
- 不同客户群体的价值画像差异很大

## 定价模型选择框架

回答以下问题找到合适的模型：

**1. 价值是否随用户数增长？**
- 是，线性增长 → 按席位
- 是，但非线性 → 创建者/查看者分离或按角色分层的席位

**2. 价值是否随使用量增长？**
- 是，可衡量 → 按用量
- 是，但使用量难以衡量 → 功能层级 + 用量上限

**3. 你的客户是追求简单的小企业吗？**
- 是 → 固定费用或简单的 2-3 级功能定价
- 否 → 跳过固定费用，选功能层级或按用量

**4. 你有需要治理/合规的企业客户吗？**
- 是 → 必须有企业套餐（即使是"联系我们"）
- 否 → 最多三个层级

**5. 这是开发者/技术产品吗？**
- 是 → 按用量或消耗制 + 免费层级是市场惯例
- 否 → 功能层级 + 固定费用更容易被接受

## 定价模型基准数据

| 指标 | 早期阶段 | 增长期 | 规模期 |
|------|---------|--------|--------|
| **试用转付费率** | 15-25% | 20-35% | 25-40% |
| **年付 vs 月付比例** | 30-50% 年付 | 40-60% 年付 | 50-70% 年付 |
| **扩展收入** | 占 MRR 的 0-10% | 10-20% | 20-40% |
| **调价频率** | 不定期 | 每年一次 | 每年一次 |
| **流失率（月度）** | 2-8% | 1-4% | 0.5-2% |

**LTV:CAC 法则：** 客户终身价值应 ≥ 3 倍获客成本。如果低于 3 倍，定价或留存（或两者）需要改进。
