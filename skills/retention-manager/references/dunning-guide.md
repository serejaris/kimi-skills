# 催款指南

支付恢复策略、重试逻辑和被动流失邮件序列。

---

## 为什么被动流失很重要

在大多数 SaaS 公司中，20-40% 的流失来自支付失败——而不是用户主动选择离开。用户并没有决定走。他们的银行卡过期了、被换发了、额度用尽了，或被银行标记了。这些情况大部分在 7-14 天内可以恢复。

**算一笔账：**
- 1,000 名活跃用户
- 3% 月度流失率 = 每月流失 30 人
- 其中 30% 为被动流失 = 每月 9 名用户因支付失败流失
- 40% 恢复率 = 每月挽回 3.6 名用户
- 按 $100 MRR 计算：每月恢复 $360，每年 $4,320——仅仅靠修复催款流程

这还没算主动流失的部分。

## 失败类型分类

支付失败不全一样。在决定重试策略前先做分类：

| 失败类型 | 拒绝代码 | 恢复方法 |
|---------|---------|---------|
| **余额不足** | `insufficient_funds` | 3-5 天后重试（余额通常会补充） |
| **银行卡过期** | `expired_card` | 先走银行卡自动更新；同时发邮件要求更新 |
| **银行卡换发** | `card_not_supported`，网络已更新 | 银行卡自动更新服务会自动处理 |
| **拒绝交易** | `do_not_honor` | 3 天后重试一次；发邮件建议联系银行 |
| **欺诈标记** | `fraudulent` | 立即发邮件；不要重试——让用户自行解决 |
| **银行卡丢失/被盗** | `lost_card`、`stolen_card` | 立即发邮件；不要重试 |
| **通用拒绝** | `generic_decline` | 7 天内重试 2 次；然后发邮件 |

**原则：** 绝不重试欺诈、丢失或被盗银行卡。重试会增加退款风险。

## 重试计划

基于卡组织研究的最优时间安排：

```
第 0 天：支付失败（初始扣费）
第 3 天：重试 1——恢复率最高（3-7 天是最佳窗口）
第 8 天：重试 2——对应月度发薪周期的余额补充
第 13 天：重试 3——最后一次自动尝试
第 16 天：取消订阅（如未恢复）
```

**Stripe 配置：**

在 Stripe Billing 设置中（Dashboard > Billing > Subscriptions and emails）：
```
Smart Retries：启用（Stripe 使用机器学习选择重试时机）
   或
手动计划：第 3 天、第 5 天、第 7 天
所有重试失败后的订阅行为：取消订阅
```

**最大化恢复的建议：**
如果使用 Smart Retries（Stripe），请禁用手动计划——两者会冲突。
Smart Retries 使用实时卡组织网络数据，通常效果优于固定计划。

## 银行卡自动更新服务

当银行换发新卡时，这些服务会自动更新银行卡信息：

| 服务商 | 服务名称 | 配置要求 |
|--------|---------|---------|
| Stripe | Account Updater | 默认启用。在 Dashboard > Settings > Card account updater 中确认 |
| Braintree | Account Updater | 需在 Control Panel > Processing > Account Updater 中启用 |
| Recurly | Account Updater | Professional 及以上套餐可用 |
| Chargebee | Smart Dunning | Chargebee 内置功能；在配置中启用 |

**预期效果：** 在催款邮件发送之前，预防 15-25% 的被动流失。

## 催款邮件序列

五封邮件。每封略微增加紧迫感。不制造负罪感——这是业务通知，不是道德审判。

---

### 邮件 1 — 第 0 天："支付失败"

**目标：** 告知情况，简化解决步骤。

```
标题：你的 [产品] 支付未成功

你好 [姓名]，

我们未能处理你 [产品] 订阅 [金额] 的支付。

这种情况偶尔会发生——银行卡过期、银行临时问题或额度限制。
很容易解决。

在这里更新支付方式：
[更新支付方式 →]

如果需要帮助，请回复此邮件。

[产品] 团队

---
支付金额：[金额]
账单日期：[日期]
下次重试：[日期 + 3 天]
```

**注意事项：**
- 在支付失败后 1 小时内发送
- 包含具体金额和日期——模糊的邮件会被忽略
- 提及下次重试日期——有些用户会等待重试看能否自动成功

---

### 邮件 2 — 第 3 天："即将重试"

**目标：** 在重试前提醒用户，让他们有机会先更新银行卡。

```
标题：[产品]——我们明天将再次尝试扣款

你好 [姓名]，

我们将于明天（[具体日期]）再次尝试你 [产品] [金额] 的支付。

如果你的银行卡信息有变，请提前更新以确保扣款成功：
[更新支付方式 →]

如果重试失败，我们会再次联系你。

[产品] 团队
```

**注意事项：**
- 在重试前一天发送，不是当天
- 简短邮件——一个目的，一个行动号召
- 如果你的支付服务商支持在银行卡更新后立即手动重试，可以在此提及

---

### 邮件 3 — 第 7 天："再次尝试——仍然失败"

**目标：** 增加紧迫感，保持温和语气，提供帮助。

```
标题：[产品] 支付仍然失败——需要你处理

你好 [姓名]，

我们已经两次尝试处理你的 [产品] 订阅支付，但都没有成功。

你的账户仍然活跃，但我们需要尽快解决此问题以避免服务中断。

一些常见的解决方法：
• 检查银行卡是否过期并更新
• 如果银行卡被意外拒绝，联系发卡行
• 如果这张卡不再可用，换一张卡

更新支付方式：
[更新支付方式 →]

仍然有困难？回复此邮件，我们帮你解决。

[产品] 团队
```

**注意事项：**
- 从通知转向解决问题
- 列出常见原因——帮助用户自行排查
- 提供人工帮助——有些用户确实遇到了困惑

---

### 邮件 4 — 第 12 天："最终通知"

**目标：** 制造紧迫感但不带威胁。清晰说明后果。

```
标题：[产品] 账户面临风险——请在 [具体日期] 前完成支付

你好 [姓名]，

我们多次尝试处理你的 [产品] 订阅支付，但一直无法扣款成功。

如果我们在 [具体日期] 前未收到支付，你的账户将被取消。

你将失去以下访问权限：
• [关键功能/数据]
• [关键功能/数据]
• 你 [X] 个月的 [数据/历史/使用记录]

这是取消前的最后一次提醒。

立即更新支付方式：
[更新支付方式 →]

需要和人沟通？[预约电话] 或直接回复此邮件。

[产品] 团队
```

**注意事项：**
- 使用具体日期——"即将"不能制造紧迫感，"3 月 15 日"可以
- 列出他们会失去什么——具体的比抽象的更有说服力
- 提供人工升级通道——这个阶段的部分流失可以通过客服人员挽回

---

### 邮件 5 — 第 16 天："账户已取消"

**目标：** 告知、留下重新激活的机会、简化恢复流程。

```
标题：你的 [产品] 账户已被取消

你好 [姓名]，

由于多次扣款未成功，我们已于今日取消你的 [产品] 订阅。
未能扣取的金额为 [金额]。

你的数据保留 90 天（至 [日期]）。

如需重新激活：
[重新激活我的账户 →]

你可以从上次中断的地方继续——所有数据完整保留。

如果你认为这是个错误，请回复此邮件，我们来帮你处理。

[产品] 团队
```

**注意事项：**
- 不要指责、不要制造负罪感——这是一封通知，不是批评
- 重新激活要无摩擦——一键操作，不是重新注册流程
- 数据保留期限给他们一个在 90 天内行动的理由

## 催款指标追踪

| 指标 | 衡量内容 | 目标 |
|------|---------|------|
| **恢复率** | 恢复的支付 / 总失败支付 | 25-40% |
| **按邮件的恢复率** | 序列中哪封邮件转化最多 | 按邮件追踪 |
| **按重试的恢复率** | 哪次重试成功最多 | 通常是重试 1（第 3 天） |
| **恢复时间** | 从首次失败到成功支付的天数 | <10 天为良好 |
| **银行卡更新命中率** | 在人工干预前自动更新的银行卡 | 占失败的 15-25% |

## 第三方催款工具

适合需要开箱即用的催款方案、无需自建的团队：

| 工具 | 适用场景 | 定价模式 |
|------|---------|---------|
| **Churnkey** | Stripe 用户，完整取消流程 + 催款 | 收入分成 |
| **ProfitWell Retain** | Stripe + Braintree，分析能力强 | 按恢复收入的百分比 |
| **Stunning** | Stripe 原生，聚焦邮件 | 月度固定费用 |
| **Recurly** | 已使用 Recurly | 内置功能 |
| **Chargebee Smart Dunning** | 已使用 Chargebee | 内置功能 |

**何时使用第三方工具：** 如果 MRR 低于 $500k 且没有工程资源自建重试逻辑和邮件序列，用工具很快就能回本。超过这个阈值，自建能获得更多控制权。
