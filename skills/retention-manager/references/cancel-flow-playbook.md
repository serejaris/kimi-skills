# 取消流程手册

设计、搭建和审查取消流程的完整参考指南。

---

## 取消流程决策树

```
用户点击"取消"或"取消订阅"
         │
         ▼
   [展示退出调查]
   "你取消的主要原因是什么？"
         │
    ┌────┴─────────────────────────────────────────┐
    │                                               │
 价格/价值                                    其他原因
    │                                               │
    ▼                                               ▼
折扣方案                                   匹配对应原因分类
(1-3 个月, 20-30%)                        (见映射表)
    │                                               │
    ▼                                               ▼
[接受?]──是──► 更新计费              [接受?]──是──► 应用方案
    │                                    │
    否                                   否
    │                                    │
    ▼                                    ▼
[确认取消]                          [确认取消]
    │                                    │
    ▼                                    ▼
[取消后页面 + 邮件]                 [取消后页面 + 邮件]
```

## 各阶段模板

### 阶段 1：取消前拦截

**触发时机：** 用户进入取消/订阅页面、点击"取消套餐"或进入账单设置。

**展示内容：** 简短的价值提醒（不是铺天盖地的挽留文案）+ "告诉我们原因"的引导。

**文案模板：**
```
标题：在你离开之前，我们想了解一下
正文：你的反馈能帮助我们改进。花 30 秒告诉我们你取消的原因
      ——也许我们有你还没试过的解决方案。
按钮：继续取消流程 →
```

**规则：**
- 不要阻断取消路径
- 每次会话只展示一次
- 移动端：单屏展示，无需滚动

---

### 阶段 2：退出调查

**设计规格：**
- 单个问题，必填
- 单选按钮（不是复选框）
- 最多 6-8 个选项
- 底部可选开放式文本框："还有什么想告诉我们的？"
- 提交后进入阶段 3——此时不要展示挽留方案

**文案模板：**
```
你取消的主要原因是什么？

○ 价格太贵，性价比不高
○ 用得不够多，不值得付费
○ 缺少我需要的功能
○ 我要换另一个工具
○ 项目或需求已经结束
○ 太复杂，不好用
○ 我只是在试用
○ 其他：[文本框]

[继续 →]
```

**数据采集：** 将原因关联到用户记录。这是你的产品反馈金矿。

---

### 阶段 3：动态挽留方案

**方案-原因映射（完整版）：**

| 选择的原因 | 主要方案 | 备选方案（如被拒绝） | 跳过方案 |
|-----------|---------|-------------------|---------|
| 太贵 | 3 个月打 7 折 | 降级套餐 | — |
| 用得不够多 | 暂停 1 个月 | 使用指导电话 | — |
| 缺少功能 | 分享功能路线图 + 替代方案 | 人工支持电话 | 功能确实不存在且近期不会有 |
| 转向竞品 | 竞品对比单页 | — | 用户已明确决定 |
| 项目结束 | 暂停 2 个月 | — | — |
| 太复杂 | 免费引导培训 | 一对一支持 | — |
| 只是试用 | — | — | 始终跳过——匹配度低，放行 |
| 其他 | 人工支持电话 | — | — |

**方案展示模板：**

```
[价格问题：]
标题：以更低的价格继续使用 [产品]
正文：我们不希望因为价格失去你。这是我们能做到的：
      接下来 3 个月享受 7 折——为你节省 [计算出的金额]。
      3 个月后恢复原价 [原始价格]。
按钮（接受）：领取我的折扣 →
按钮（拒绝）：不了，继续取消 →

[用得不够多：]
标题：60 天免费——暂停你的账户
正文：生活总会忙起来。你可以将 [产品] 暂停最多 60 天。
      你的数据完整保留，随时可以恢复。暂停期间不收费。
按钮（接受）：暂停我的账户 →
按钮（拒绝）：不了，继续取消 →
```

**方案规则：**
- 每次取消操作只展示一个方案——绝不同时展示多个
- 如果用户拒绝，直接进入阶段 4
- 如果用户在 30 天内再次取消，不要展示相同方案
- 记录展示了哪个方案以及是否被接受

---

### 阶段 4：取消确认

**应包含的内容：**
- 明确说明将会发生什么
- 访问权限截止日期（具体日期，不是"计费周期结束时"）
- 数据保留政策（数据保留多久）
- 联系客服（以防用户改变主意）
- 文案清晰的确认按钮

**文案模板：**
```
你的订阅将被取消。

接下来会发生：
• 你的访问权限持续到 [具体日期]
• 取消后你的数据保留 90 天
• 90 天后你的账户数据将被删除
• 你可以在 [90 天截止日期] 之前随时重新激活

如果你改变主意，请联系 [邮箱] 或访问 [重新激活链接] 重新激活。

[确认取消]    [返回]
```

---

### 阶段 5：取消后序列

**即时：取消确认邮件**

```
标题：你的 [产品] 订阅已取消

你好 [姓名]，

你的 [产品] 订阅已按要求取消。

接下来会发生：
- 访问权限持续到 [日期]
- 你的数据保留 90 天（至 [日期]）
- 如需重新激活，请访问：[重新激活链接]

如果这是误操作或你有任何问题，请回复此邮件或访问 [客服链接]。

[产品] 团队
```

**第 7 天：再互动邮件**

```
标题：你的 [产品] 账户还在

你好 [姓名]，

你取消已经一周了。你的账户和数据在 [日期] 之前仍完整保留。

如果情况有变，你可以一键重新激活——无需重新设置。

[重新激活我的账户 →]

没有压力——只是想让你知道，门随时为你敞开。

[产品] 团队
```

**第 30 天：召回邮件（仅在有产品更新或相关优惠时发送）**

```
标题：[产品] 更新：[用户提到的具体功能或相关改进]

你好 [姓名]，

自你离开后，我们上线了 [与其取消原因相关的具体更新]。
[2-3 句描述发生了什么变化。]

如果 [他们提到的具体问题] 是你离开的原因，也许值得再看看。

[看看有什么新的 →]    或    [重新激活 →]

[产品] 团队
```

## 取消流程审查评分卡

对现有流程的每个维度打分（0-10）：

| 维度 | 0 分 | 5 分 | 10 分 |
|------|------|------|-------|
| **可达性** | 取消需提交工单 | 在设置中但位置隐蔽 | 在账单设置中清晰可见 |
| **退出调查** | 无 | 选填、多个问题 | 必填、单个问题、映射到方案 |
| **挽留方案** | 无或笼统折扣 | 有方案但未映射 | 方案与退出原因对应 |
| **确认清晰度** | 表述模糊 | 提到了权限截止日期 | 明确日期、数据政策、重新激活路径 |
| **取消后序列** | 无 | 一封通用邮件 | 即时确认 + 第 7 天再互动 |
| **催款** | 无 | 仅基础重试 | 重试 + 邮件序列 + 银行卡更新 |
| **分析** | 无追踪 | 基础取消计数 | 原因追踪、挽留率、恢复率 |

**评分解读：**
- 60-70：基础扎实。修复 0-5 分的维度。
- 40-59：有明显的收入流失。优先完善调查 + 方案映射。
- <40：巨大机会。使用本手册从零搭建。

## 平台实施说明

### Stripe
- 使用 Customer Portal 搭建取消流程（可通过 Stripe Dashboard 自定义）
- 启用 Stripe Billing webhooks：`customer.subscription.deleted`、`invoice.payment_failed`
- Stripe Radar 帮助区分机器人发起的失败和真实的银行卡失败
- Account Updater：大多数套餐默认启用——在 Dashboard > Settings 中确认

### Chargebee / Recurly
- 两者都有原生取消流程构建器，支持收集退出原因
- 催款序列可在产品内配置
- 通过 webhook 连接邮件服务商（Intercom、Customer.io 等）

### 自建计费系统
- 将取消流程构建为独立路由（不要内嵌在设置页中）
- 为每个用户存储 `cancel_reason`、`save_offer_shown`、`save_offer_accepted`
- 重试逻辑：使用延迟队列的后台任务实现
