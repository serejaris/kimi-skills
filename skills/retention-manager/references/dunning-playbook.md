# 催款手册

恢复失败支付和降低被动流失的完整指南。

---

## 为什么催款很重要

- 支付失败造成 30-50% 的订阅流失
- 大多数失败支付通过正确策略是可以恢复的
- 订阅制企业每年因被动流失损失约 1290 亿美元
- 有效的催款能恢复 50-60% 的失败支付

---

## 催款时间线

```
第 -30 至 -7 天：预防阶段（预防支付失败）
第 0 天：        支付失败 → 智能重试 #1 + 邮件 #1
第 1-3 天：      智能重试 #2 + 邮件 #2
第 3-5 天：      智能重试 #3
第 5-7 天：      智能重试 #4 + 邮件 #3
第 7-10 天：     最终重试 + 邮件 #4（最终警告）
第 10-14 天：    宽限期结束 → 账户暂停/取消
第 14 天以后：   召回序列启动
```

## 预防阶段：在失败发生前预防

### 银行卡到期管理

| 时间 | 操作 |
|------|------|
| 到期前 30 天 | 邮件："你尾号 4242 的银行卡下个月到期" |
| 到期前 15 天 | 邮件："更新支付方式以避免服务中断" |
| 到期前 7 天 | 邮件："你的银行卡 7 天后到期——立即更新" |
| 到期前 3 天 | 应用内横幅："支付方式即将到期" |

**邮件模板——银行卡即将到期：**
```
标题：你尾号 4242 的银行卡即将到期

你好 [姓名]，

你的 [产品] 订阅绑定的银行卡将于 [日期] 到期。

立即更新支付方式以避免服务中断：

[更新支付方式 →]

只需 30 秒。

— [产品] 团队
```

### 银行卡自动更新服务

主要卡组织提供自动银行卡更新服务：

| 服务 | 卡组织 | 功能 |
|------|--------|------|
| Visa Account Updater (VAU) | Visa | 自动更新存储的卡号和有效期 |
| Mastercard Automatic Billing Updater (ABU) | Mastercard | Mastercard 的同类服务 |
| Amex Cardrefresher | American Express | Amex 的同类服务 |

**效果：** 将因过期/换发银行卡导致的硬拒绝减少 30-50%。

**启用方式：**
- **Stripe**：自动启用——默认开启
- **Chargebee**：通过网关设置启用
- **Recurly**：内置功能，默认启用
- **Braintree**：联系支付处理商启用

### 备用支付方式

引导用户添加第二种支付方式：
- 注册时："添加一个备用支付方式"（转化率低）
- 首次支付成功后："用备用银行卡保护你的账户"（时机更好）
- 恢复失败支付后："添加备用方式以防止未来中断"（最佳时机——他们刚经历过失败的痛苦）

### 续费前通知

针对年度套餐或高价值订阅：
- 续费前 7 天发邮件，包含金额和日期
- 附带更新支付方式的链接
- 展示续费包含的内容
- 部分地区法规要求自动续费必须提前通知

## 智能重试策略

### 拒绝类型分类

| 代码 | 类型 | 含义 | 是否重试？ |
|------|------|------|-----------|
| `insufficient_funds` | 软拒绝 | 余额暂时不足 | 是——2-3 天后重试 |
| `card_declined`（通用） | 软拒绝 | 各种临时原因 | 是——重试 3-4 次 |
| `processing_error` | 软拒绝 | 网关/网络问题 | 是——24 小时内重试 |
| `expired_card` | 硬拒绝 | 银行卡已过期 | 否——请求更换新卡 |
| `stolen_card` | 硬拒绝 | 银行卡被挂失 | 否——请求更换新卡 |
| `do_not_honor` | 软/硬 | 银行拒绝（模糊） | 再试一次，然后请求更换新卡 |
| `authentication_required` | 认证 | 需要 SCA/3DS 验证 | 引导用户完成验证 |

### 各服务商的重试计划

**Stripe（Smart Retries——推荐）：**
- 在 Stripe Dashboard → Billing → Settings 中启用 "Smart Retries"
- Stripe 的机器学习模型基于数十亿笔交易数据选择最优重试时机
- 通常在 3-4 周内重试 4-8 次
- 比固定计划的恢复率高约 15%

**手动重试计划（无智能重试时使用）：**

| 重试次数 | 时间 | 最佳日期/时间 |
|---------|------|-------------|
| 1 | 第 1 天（失败后 24 小时） | 上午，与原始扣费同一天 |
| 2 | 第 3 天 | 换个时间段尝试 |
| 3 | 第 5 天 | 常见发薪日后（1 号、15 号） |
| 4 | 第 7 天 | 下一个工作日上午 |
| 5（最终） | 第 10 天 | 宽限期结束前最后一次尝试 |

**重试时机要点：**
- 在原始支付成功的同一天重试
- 在常见发薪日（每月 1 号和 15 号）之后重试
- 避免周末重试（通过率较低）
- 上午重试（当地时间 8-10 点）效果略好

## 宽限期管理

### 宽限期内的处理

| 设置项 | 建议 |
|--------|------|
| 时长 | 最终重试后 7-14 天 |
| 访问权限 | 降级（只读）或完整访问 |
| 展示 | 应用内横幅："支付逾期——更新支付方式以继续使用" |
| 重试 | 宽限期内继续后台重试 |
| 沟通 | 催款邮件继续发送 |

### 访问权限降级方案

**方案 A：宽限期内保持完整访问（推荐用于 B2B）**
- 摩擦更低，用户感受到尊重
- 恢复率更高（他们仍能感受到价值）
- 风险：部分用户可能利用宽限期

**方案 B：只读访问（推荐用于 B2C）**
- 可以查看但不能创建/编辑
- 制造紧迫感但不会产生数据丢失的恐惧
- 明确提示："更新支付方式以恢复完整访问"

**方案 C：立即锁定（不推荐）**
- 过于激进，损害客户关系
- 恢复率更低
- 仅适用于极低价套餐

### 宽限期后

| 时间 | 操作 |
|------|------|
| 宽限期结束 | 暂停账户（不是删除） |
| 暂停后第 1 天 | 发送"你的账户已暂停"邮件 |
| 暂停后第 7 天 | 发送"你的数据还在"提醒 |
| 暂停后第 30 天 | 尝试召回，附带新优惠 |
| 暂停后第 60 天 | 最终召回尝试 |
| 暂停后第 90 天 | 数据删除警告（如适用） |

## 各服务商配置指南

### Stripe

**启用 Smart Retries：**
1. Dashboard → Settings → Billing → Subscriptions and emails
2. 在重试规则下启用 "Smart Retries"
3. 在 Dashboard → Settings → Emails 中设置支付失败邮件

**自定义重试规则（不使用 Smart Retries 时）：**
```
重试 1：失败后第 3 天
重试 2：失败后第 5 天
重试 3：失败后第 7 天
最终：  最后一次重试后将订阅标记为未付
```

**需处理的 Webhook 事件：**
- `invoice.payment_failed` — 触发催款
- `invoice.paid` — 取消催款，恢复访问
- `customer.subscription.updated` — 状态变更
- `customer.subscription.deleted` — 最终取消

### Chargebee

**内置催款：**
1. Settings → Configure Chargebee → Retry Settings
2. 配置重试次数和间隔
3. Settings → Configure Chargebee → Email Notifications → Dunning

**催款选项：**
- 可配置计划的自动重试
- 内置催款邮件（可自定义模板）
- 按套餐配置宽限期

### Paddle

**托管催款：**
- Paddle 自动处理重试和催款
- 自定义程度有限（Paddle 管理客户关系）
- Webhook：`subscription.payment_failed`、`subscription.cancelled`
- 适合免运维方案

### Recurly

**收入恢复：**
1. Configuration → Dunning Management
2. 按套餐设置重试计划
3. 配置宽限期和最终操作（暂停 vs 取消）

**高级功能：**
- 机器学习优化重试时机
- 按套餐定制催款计划
- 内置 Account Updater

## 应用内催款

不要只依赖邮件。在应用中展示支付失败提示：

### 横幅模式
```
┌──────────────────────────────────────────────────────┐
│ ⚠ 你 ¥29 的支付失败。更新银行卡以避免失去访问权限。     │
│ [更新支付方式 →]  [关闭]                                │
└──────────────────────────────────────────────────────┘
```

**规则：**
- 催款期间每次页面加载都展示
- 允许关闭（但下次会话再次展示）
- 直接链接到支付更新页面（点击次数越少越好）
- 不要阻断产品使用——让用户继续使用

### 弹窗模式（最终警告时使用）
```
┌─────────────────────────────────────┐
│                                     │

│                                     │
│  更新支付方式以保留你的                │
```

## 衡量催款效果

### 关键指标

| 指标 | 计算方式 | 目标 |
|------|---------|------|
| 恢复率 | 恢复的支付 / 总失败支付 | 50-60% |
| 按拒绝类型的恢复率 | 每种类型的恢复 / 失败 | 软拒绝：70%+，硬拒绝：40%+ |
| 恢复时间 | 从失败到成功支付的天数 | <5 天 |
| 预防阶段预防率 | 预防的失败 / 预期失败 | 20-30% |
| 催款邮件打开率 | 打开数 / 发送数（按邮件） | 60%+ |
| 催款邮件点击率 | 点击数 / 打开数（按邮件） | 30%+ |
| 月度恢复收入 | 恢复支付的总金额 | 追踪趋势 |
| 被动流失损失收入 | 失败 + 未恢复支付的总金额 | 追踪趋势 |

### 基准对比

**按公司阶段：**

| 阶段 | 典型被动流失 | 优化后目标 |
|------|------------|-----------|
| 早期（< 100 万美元 ARR） | MRR 的 3-5%/月 | 1-2% |
| 成长期（100-1000 万美元 ARR） | MRR 的 2-4%/月 | 0.5-1.5% |
| 规模化（1000 万美元+ ARR） | MRR 的 1-3%/月 | 0.3-0.8% |

### 投资回报计算

```
月度失败支付 MRR：         $10,000
当前恢复率：               30%（恢复 $3,000）
目标恢复率：               60%（恢复 $6,000）
月度改善：                 $3,000/月
年度改善：                 $36,000/年
催款优化成本：             约 $200-500/月（工具费）
投资回报率：               6-15 倍
```
