# 参考资料

## 依赖类别

评估候选模块是否适合深化时，需对其依赖进行分类：

### 1. 进程内依赖

纯计算、内存状态，无 I/O。始终可以深化——直接合并模块并测试即可。

### 2. 本地可替代依赖

拥有本地测试替代品的依赖（例如用 PGLite 替代 Postgres、内存文件系统）。如果测试替代品存在，就可以深化。深化后的模块在测试套件中使用本地替代品进行测试。

### 3. 远程但自有依赖（端口与适配器）

跨网络边界的自有服务（微服务、内部 API）。在模块边界定义端口（接口）。深模块拥有业务逻辑；传输层通过注入方式提供。测试使用内存适配器，生产环境使用真实的 HTTP/gRPC/队列适配器。

推荐表述："定义共享接口（端口），为生产环境实现 HTTP 适配器，为测试实现内存适配器，这样即使部署跨越网络边界，业务逻辑仍然可以作为一个深模块来测试。"

### 4. 真正的外部依赖（Mock）

你无法控制的第三方服务（Stripe、Twilio 等）。在边界处进行 Mock。深化后的模块将外部依赖作为注入的端口，测试提供 Mock 实现。

## 测试策略

核心原则：**替代，而非叠加。**

- 浅模块的旧单元测试在边界测试建立后就是浪费——删除它们
- 在深化模块的接口边界编写新测试
- 测试通过公开接口断言可观测的结果，而非内部状态
- 测试应能在内部重构后依然通过——它们描述的是行为，不是实现

## Issue 模板

<issue-template>

## 问题

描述架构摩擦：

- 哪些模块是浅的且紧耦合的
- 模块之间的接缝存在哪些集成风险
- 为什么这使得代码库更难以导航和维护

## 提议的接口

选定的接口设计：

- 接口签名（类型、方法、参数）
- 使用示例，展示调用方如何使用
- 内部隐藏了哪些复杂度

## 依赖策略

适用哪种类别以及如何处理依赖：

- **进程内**：直接合并
- **本地可替代**：使用 [具体替代品] 测试
- **端口与适配器**：端口定义、生产适配器、测试适配器
- **Mock**：外部服务的 Mock 边界

## 测试策略

- **需要编写的新边界测试**：描述在接口处需要验证的行为
- **需要删除的旧测试**：列出将变得冗余的浅模块测试
- **测试环境需求**：所需的本地替代品或适配器

## 实施建议

持久的架构指导，不与当前文件路径绑定：

- 模块应该负责什么（职责）
- 应该隐藏什么（实现细节）
- 应该暴露什么（接口契约）
- 调用方应该如何迁移到新接口
